# 需求文档 - 数据权限过滤功能

## 1. 需求描述

### 1.1 业务背景
当前系统所有用户都可以看到完整的用户列表、角色列表和活动日志，缺乏数据权限隔离机制。需要实现基于角色的数据权限过滤，确保用户只能看到和查询与自身角色相关的数据。

### 1.2 功能目标
- 用户列表：根据当前登录用户的角色，过滤显示该角色下的用户
- 角色列表：根据当前登录用户的角色层级，过滤显示有权限查看的角色
- 活动日志：根据当前登录用户的角色，过滤显示该角色相关的操作日志

### 1.3 使用场景
- **管理员角色**：可以查看所有用户、角色和活动日志
- **普通角色**：只能查看本角色下的用户和相关操作日志
- **部门角色**：只能查看本部门/本角色范围内的数据

---

## 2. 功能规格

### 2.1 用户列表过滤

| 功能点 | 描述 |
|--------|------|
| 数据过滤 | 根据当前用户角色ID，过滤用户列表，只显示该角色关联的用户 |
| 搜索过滤 | 搜索时也需遵循数据权限，只能搜索本角色下的用户 |
| 分页支持 | 过滤后的数据支持分页展示 |

### 2.2 角色列表过滤

| 功能点 | 描述 |
|--------|------|
| 角色层级 | 根据角色层级关系进行过滤，上级角色可查看下级角色 |
| 平级隔离 | 同层级角色之间不可互相查看 |
| 管理员豁免 | 超级管理员可查看所有角色 |

### 2.3 活动日志过滤

| 功能点 | 描述 |
|--------|------|
| 操作人过滤 | 只显示当前角色下用户的操作日志 |
| 时间范围 | 支持按时间范围筛选 |
| 操作类型 | 支持按操作类型筛选 |

---

## 3. 权限需求

### 3.1 角色权限矩阵

| 角色类型 | 查看所有用户 | 查看本角色用户 | 查看所有角色 | 查看本角色 | 查看所有日志 | 查看本角色日志 |
|---------|------------|---------------|-------------|-----------|-------------|----------------|
| 超级管理员 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 部门管理员 | ❌ | ✅ | ❌ | ✅ | ❌ | ✅ |
| 普通角色 | ❌ | ✅ | ❌ | ✅ | ❌ | ✅ |

### 3.2 访问控制规则

1. **超级管理员**（admin）：不受数据权限限制，可查看所有数据
2. **非管理员角色**：只能查看与自身角色关联的数据
3. **角色层级**：上级角色可查看下级角色及其用户，但下级角色不可查看上级

## 5. API接口设计

### 5.1 用户列表接口

**路径**: `GET /user/list`

**请求参数**:

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | int | 否 | 页码，默认1 |
| page_size | int | 否 | 每页数量，默认10 |
| username | string | 否 | 用户名搜索 |

**后端处理逻辑**:
1. 获取当前登录用户信息
2. 如果是超级管理员，查询所有用户
3. 如果是非管理员，根据用户的角色ID查询关联用户

**响应格式**:

```json
{
  "code": 0,
  "data": {
    "users": [],
    "total": 0
  },
  "message": "success"
}
```

### 5.2 角色列表接口

**路径**: `GET /role/list`

**请求参数**:

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | int | 否 | 页码，默认1 |
| page_size | int | 否 | 每页数量，默认10 |
| name | string | 否 | 角色名称搜索 |

**后端处理逻辑**:
1. 获取当前登录用户信息
2. 如果是超级管理员，查询所有角色
3. 如果是非管理员，查询当前角色及其子角色

**响应格式**:

```json
{
  "code": 0,
  "data": {
    "roles": [],
    "total": 0
  },
  "message": "success"
}
```

### 5.3 活动日志列表接口

**路径**: `GET /activity/list`

**请求参数**:

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | int | 否 | 页码，默认1 |
| page_size | int | 否 | 每页数量，默认10 |
| start_time | string | 否 | 开始时间 |
| end_time | string | 否 | 结束时间 |

**后端处理逻辑**:
1. 获取当前登录用户信息
2. 如果是超级管理员，查询所有日志
3. 如果是非管理员，根据用户角色查询该角色下用户的日志

**响应格式**:

```json
{
  "code": 0,
  "data": {
    "activities": [],
    "total": 0
  },
  "message": "success"
}
```

## 6. 实现方案

### 6.1 后端实现

#### 6.1.1 中间件层
- 创建数据权限中间件 `data_permission.go`
- 在请求处理前获取当前用户角色信息
- 将角色信息存入上下文，供后续逻辑使用

#### 6.1.2 逻辑层修改

**用户列表逻辑** (`listuserlogic.go`):
```go
// 根据角色过滤用户
if !isAdmin {
    // 查询当前用户角色下的所有用户
    users, total, err := l.svcCtx.UserModel.ListByRoleID(roleID, page, pageSize)
} else {
    // 超级管理员查询所有用户
    users, total, err := l.svcCtx.UserModel.List(page, pageSize)
}
```

**角色列表逻辑** (`listrolelogic.go`):
```go
// 根据角色层级过滤
if !isAdmin {
    // 查询当前角色及其子角色
    roles, total, err := l.svcCtx.RoleModel.ListByParentID(roleID)
} else {
    // 超级管理员查询所有角色
    roles, total, err := l.svcCtx.RoleModel.List(page, pageSize)
}
```

**活动日志逻辑** (`listactivitylogic.go`):
```go
// 根据角色过滤日志
if !isAdmin {
    // 查询当前角色下用户的日志
    activities, total, err := l.svcCtx.ActivityModel.ListByRoleID(roleID, page, pageSize)
} else {
    // 超级管理员查询所有日志
    activities, total, err := l.svcCtx.ActivityModel.List(page, pageSize)
}
```

#### 6.1.3 模型层新增方法

**用户模型** (`user_model.go`):
```go
func (m *UserModel) ListByRoleID(roleID uint, page, pageSize int) ([]User, int64, error)
```

**角色模型** (`role_model.go`):
```go
func (m *RoleModel) ListByParentID(parentID uint) ([]Role, int64, error)
```

**活动日志模型** (`activity_model.go`):
```go
func (m *ActivityModel) ListByRoleID(roleID uint, page, pageSize int) ([]Activity, int64, error)
```

## 9. 注意事项

1. **性能考虑**：数据权限过滤可能影响查询性能，需添加适当索引
2. **缓存策略**：考虑缓存用户角色信息，减少数据库查询
3. **日志记录**：记录数据权限过滤的操作日志，便于审计
4. **API兼容性**：保持现有API接口不变，后端透明处理数据过滤

## 2. 测试目标

### 2.1 总体目标
确保 ERP 进销存系统的各项功能符合需求规格，保证系统的正确性、稳定性、安全性和易用性。

### 2.2 具体目标
- 验证采购订单全流程（创建→审核→入库→完成）的正确性
- 验证销售订单全流程（创建→审核→出库→完成）的正确性
- 验证库存管理功能的准确性和数据一致性
- 验证产品、供应商、客户的 CRUD 操作正确性
- 验证系统异常处理和错误提示的合理性
- 验证系统性能和响应时间满足业务需求

---

## 3. 测试范围

### 3.1 功能测试范围

| 模块 | 功能点 |
|------|--------|
| **采购管理** | 采购订单创建、审核、入库、取消、删除、查询 |
| **销售管理** | 销售订单创建、审核、出库、取消、删除、查询 |
| **库存管理** | 库存查询、库存调整、库存变动记录、库存预警 |
| **产品管理** | 产品创建、查询、更新、删除、分类管理 |
| **供应商管理** | 供应商创建、查询、更新、删除 |
| **客户管理** | 客户创建、查询、更新、删除 |
| **系统管理** | 用户登录、权限验证、菜单管理 |

### 3.2 非功能测试范围
- 接口响应时间测试
- 系统稳定性测试（连续运行72小时）
- 数据一致性测试
- 安全性测试（权限控制、SQL注入防护）

## 4. 测试策略

### 4.1 测试层次

```
┌─────────────────────────────────────────────┐
│           端到端测试 (E2E)                   │
│   完整业务流程测试（前端+后端+数据库）         │
├─────────────────────────────────────────────┤
│           集成测试 (Integration)            │
│   接口联调测试（多个模块协同工作）            │
├─────────────────────────────────────────────┤
│           单元测试 (Unit)                   │
│   单个模块/函数独立测试                      │
└─────────────────────────────────────────────┘
```

### 4.2 测试方法

| 测试类型 | 方法 | 工具 |
|----------|------|------|
| 单元测试 | 白盒测试 | Go Testing + Mock |
| 集成测试 | 灰盒测试 | Postman / curl |
| 端到端测试 | 黑盒测试 | Cypress / Playwright |
| 性能测试 | 压力测试 | JMeter / k6 |

### 4.3 测试覆盖策略

1. **等价类划分**：将输入数据划分为有效等价类和无效等价类
2. **边界值分析**：测试边界条件（最小值、最大值、空值、零值）
3. **状态转换测试**：测试订单状态流转的正确性
4. **异常路径测试**：测试错误输入、异常场景的处理

## 9. 测试风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 测试数据准备不足 | 测试覆盖率低 | 提前准备测试数据脚本 |
| 接口变更频繁 | 测试用例维护成本高 | 采用契约测试，保持接口稳定性 |
| 环境不稳定 | 测试结果不可靠 | 配置独立测试环境 |
| 时间紧迫 | 测试不充分 | 优先测试核心功能 |

## 10. 质量标准

### 10.1 测试通过标准

| 指标 | 标准 |
|------|------|
| 单元测试覆盖率 | ≥ 80% |
| 功能测试通过率 | ≥ 95% |
| 严重缺陷数 | 0 |
| 中等缺陷数 | ≤ 3 |
| 轻微缺陷数 | ≤ 5 |
| 接口响应时间 | ≤ 500ms（P95） |

### 10.2 缺陷严重程度定义

| 级别 | 定义 | 处理优先级 |
|------|------|------------|
| **高** | 影响核心功能，导致系统不可用或数据丢失 | 立即处理 |
| **中** | 影响部分功能，用户可通过其他方式操作 | 24小时内处理 |
| **低** | 界面显示问题或不影响功能的小问题 | 下个迭代处理 |
